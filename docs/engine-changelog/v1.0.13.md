The `renderFieldList` retirement — the last step of the card-system redesign (v1.0.10). Every fleet-wide call site had already migrated to the `card-helpers` vocabulary (CHI #172/#173, NYC #68, SF #34), leaving the legacy path as dead code; this release removes it. Changed blocks only — **no new blocks, no fence-set change, no fork adoption required**.

- **Changed `render-helper`**: `renderFieldList` (the legacy `dt`/`dd` row builder) is removed. The fence is kept as a documented tombstone comment rather than deleted outright — the artifact builder rejects an empty fence pair, and removing the fence would force a cross-fork fence-set change (sibling pre-clean PRs) for no functional gain. The block goes from the function body to an 8-line comment.
- **Changed `styles-app`**: the `.result-row` / `dl.result-fields` CSS (the legacy table's styling) is removed.
- **Changed `polygon-factory`, `polygonCountyEntry`**: the `hasHtml ? renderFieldList(rows) : renderFieldStack(rows)` seam and the `f.html` field-building branch are gone — the render always returns `renderFieldStack(rows)`. No registration passes `html` fields anywhere in the fleet, so this is pure dead-code removal.
- **Changed `school-zone-factory`**: the `legacyHtml` path (the two `renderFieldList` branches for the caller-HTML `profileHtml`/`moreHtml` opts) is removed; the redesigned data-only card (`profileLink`/`moreLink` + the §6a grade-range pill) is the only path now.
- **Changed `exports`**: `renderFieldList` is dropped from the debug namespace (`window.<Metro>Explorer`).
- Consume-side scripts are unchanged.
- **Fork adoption: none.** This is a changed-blocks-only release — the bump splices the new bodies in atomically (the `renderFieldList` definition and its debug export disappear in the same apply, so there is never an intermediate state where the export references an undefined function). Any fork smoke test that still exposed `renderFieldList` via the debug namespace should drop that reference, but none in the fleet asserts on it.

**Why a tombstone, not a deletion:** `apply_engine.py` requires each fork's fenced-block set to match the manifest exactly (it fails on a fork carrying a fence the release lacks). Truly deleting the `render-helper` fence would therefore need a pre-clean PR in every sibling to drop its own fence first — the mirror of v1.0.10's fence-seeding, with a transient window. Tombstoning the fence keeps the block set stable so this ships as an ordinary changed-blocks release, exactly like v1.0.11/v1.0.12. `docs/ENGINE_SYNC.md` records this as the fleet's convention for retiring a shared helper.
