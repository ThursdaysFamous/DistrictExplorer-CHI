Graceful "data source unavailable" cards (`mod.dataSource`) ‚Äî an additive engine-contract-field change, the same shape as v1.0.4's `coverage` (bundle sha `5e99b532‚Ä¶cefa452`). A layer that declares no `dataSource` takes exactly the pre-v1.0.5 code path, so a fork that declares nothing (NYC today) is behaviorally unchanged.

- **New optional layer-contract field `dataSource = { label, lookup?: { url, text, label } }`** ‚Äî declares the live data source a layer fetches from at runtime. When that source is unreachable and the layer's card would otherwise show the generic error, the engine instead renders a calm, honest degradation: it names the source (`label`), states plainly the outage is on the data provider's end ‚Äî not this tool ‚Äî and doesn't affect the other layers, links an official self-service lookup when `lookup` is supplied, and offers the same Retry. A layer that declares nothing shows the generic red "couldn't load / didn't respond" error exactly as before.
- Engine surface (in `overlay-cards`): `setCardError` branches on `mod.dataSource`; a new `renderSourceUnavailable(mod, body)` helper builds the graceful card from the declaration, reusing the shared `renderFieldList` + `sanitize` helpers and the same `onLayerToggled`-based Retry as the generic error card. Both failure paths route through `setCardError` ‚Äî the overlay-load `.catch` (a boundary fetch that fails, even before a point is picked) and the query `.catch` ‚Äî so a single declaration covers a boundary-load failure and a query failure alike. Metro-agnostic: every city-specific string (the provider name, the official-lookup URL/label) comes from the layer's own declaration, so the block stays byte-identical across forks; the calm notice uses the engine-referenced `--slate` palette var, not the red `--err` of `state-error`.
- No new factory plumbing: because most layers register through the shared ENGINE factories (whose object literals a fork can't edit), a fork attaches `dataSource` to the registered module *after* registration; the engine only reads `mod.dataSource`, however it got set. The field is never mutated by the engine.
- Consume-side scripts (`apply_engine.py`, `check_engine_parity.py`, `generate_metro_files.py`, `metro-worksheet.schema.json`) are unchanged.
- Fork-side reference (travels in the CHI repo, not the assets): CHI declares `dataSource` on its nine City-of-Chicago-Data-Portal layers ‚Äî ward, ward-precinct, community-area, fire-station, the three CPS attendance zones, and the two CPS networks ‚Äî all labelled "The City of Chicago Data Portal", with an official-lookup link on ward (chicago.gov ward-and-alderperson finder) and the three CPS zones (CPS "Find a School"); ward additionally keeps its roster-best-effort query, so a roster-only outage still reports the ward number rather than degrading. Its smoke test asserts the generic error path (a non-`dataSource` `data/app` layer still shows `state-error` + Retry) is unchanged. Forks adopt by declaring `dataSource` on their own live-fetch layers.
