Card-order conformance fixes from the fleet-wide card audit (2026-07-20) — two behavior-tightening changes inside existing blocks, no new contract fields.

- **`cps-network-factory`**: the render pushed contact before location (`Phone` above `Office Address`), inverting the fleet card convention (identifier → representative → location → contact → link). The two rows are swapped; the address row keeps its map-pin affordance. Affects any fork registering through this factory (today: Chicago's two CPS network layers; NYC/SF register none).
- **`chamber-factory`**: the profile-link *label* keyed off `member` existence rather than `memberUrl` — a roster entry with no `url` field produced the chamber-directory fallback link labeled as the member's own page ("Senator's page" / "Member profile"). The label now follows the actual href: only a roster-supplied, `safeHttpUrl`-validated member URL earns the per-member label; the directory fallback is labeled as the directory. Chambers whose rosters carry `url` for every member (CHI ILGA, SF CA, congress in every fork) render identically before and after; NY's state chambers (whose roster currently ships no `url`) stop mislabeling the fallback the moment this lands — and light up per-member labels automatically when their roster gains `url` fields.
- Consume-side scripts (`apply_engine.py`, `check_engine_parity.py`, `generate_metro_files.py`, `metro-worksheet.schema.json`) are unchanged.
- Fork adoption: nothing to do — both changes are inside shared factories. Fork-side roster builders that want the per-member label should emit a `url` field per member (see the card convention in each fork's CLAUDE.md).
