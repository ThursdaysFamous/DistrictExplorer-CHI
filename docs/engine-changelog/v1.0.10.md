The card-system redesign (docs/CARD_RENDER_API.md; design refs in docs/design_handoff_county_board_card/ + docs/design_handoff_card_system/). Two new ENGINE blocks and seven changed blocks; every change is additive — `renderFieldList` and the legacy `dt`/`dd` rendering keep serving unmigrated call sites, so a fork adopts the new vocabulary at its own pace.

- **New block `card-helpers`**: the shared row vocabulary — `renderBodyIntro`, `renderPersonRows` (role/party badges, hometown notes, committees `<details>` expanders, tinted countywide variant), `renderSectionLabel`, `renderOfficeGroup`, `renderNearestRows`, `renderLinkRow` (title/note/phone/email/links), `renderFieldStack` (the generic label-over-value stack). Data-only by contract: no `html` inputs exist on this surface; every external URL is scheme-checked through `safeHttpUrl` inside the helpers, and `mailto:`/`tel:` hrefs are helper-built (an email address is never printed; phone displays use non-breaking hyphens while the `tel:` href keeps dialable characters only).
- **New block `styles-card-v2`**: card chrome (white, 3px accent, radius 10, shadow), the handoff palette scoped to `.layer-block`, restyled loading/error/empty states (edge accents via `:has`), the full row-vocabulary CSS, the shell footer, and container-query mobile deltas at ~420px of the cards column. Additive overrides — the legacy `.result-row` rules in `styles-app` are untouched.
- **Changed `overlay-cards`**: presence-based per-result extras — `cardIdentifier(result)` renders a header identifier pill, `primaryLink(result)` a shell-owned footer link, `compact: true` the 4b name-only header (value + `cardMeta` in the head; `render()` skipped on success). Every non-result state clears the extras. Loading/empty copy updated per the handoff's 4e; `renderSourceUnavailable`'s lookup row now uses `renderLinkRow`. A module declaring none of the new fields renders exactly as before.
- **Changed `relationship-pinning`**: the pin control relocates into the shell footer when one exists (and creates it for a flush card with no `primaryLink`); its polygon-only/containment rules are unchanged.
- **Changed `chamber-factory`**: renders the 4a representative card (party badge, offices behind one desktop-open `<details>`, scraped office lines split so the first phone-like non-fax line becomes the `tel:` link); district identity moves to the pill; directory/profile to the footer.
- **Changed `nearest-point-factory`**: 4d rows with distance pills. New optional `opts.item(props) → {name, note}` for the two-line split; `opts.line` remains the fallback, so existing registrations are untouched.
- **Changed `polygon-factory`**: new opts passthroughs — `compact`/`compactMetaField` (4b header) and `primaryLink` (footer link). The generic path renders the new `renderFieldStack` **only for data-only field specs**; a spec still carrying `html` fields keeps the legacy `dt`/`dd` card byte-for-byte.
- **Changed `school-zone-factory`**: new data-only opts `profileLink(id) → {label, url}` and `moreLink` (link renders in the footer). A fork still supplying `profileHtml`/`moreHtml` keeps the legacy card unchanged.
- **Changed `cps-network-factory`**: chief as a badged person row, the network office as an office block, network number in the pill.
- Consume-side scripts (`apply_engine.py`, `check_engine_parity.py`, `generate_metro_files.py`, `metro-worksheet.schema.json`) are unchanged.
- Fork adoption: nothing is required — the bump restyles card chrome/states and the factory-driven cards (chambers, nearest lists, CPS-schema layers with data-only opts) while every fork-local `renderFieldList` card keeps rendering inside the new chrome (the accepted transitional look; CHI's own migration is the reference: PRs #172/#173). Two porting notes: (1) a fork whose `smoke_test.mjs` asserts factory-card text should adopt CHI's pill-aware card reader — the district identifier now lives in the header `.card-id-pill`, outside the card body; (2) when migrating fork cards, move links to `primaryLink`/data opts rather than `html` fields — the legacy branches are scheduled for removal once the fleet-wide `renderFieldList` grep hits zero (docs/CARD_RENDER_API.md).
