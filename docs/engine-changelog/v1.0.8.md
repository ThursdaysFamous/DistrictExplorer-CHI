Robust office-pin address cleaning — fixes the recurring "map pin doesn't drop when a card is selected" bug at its root, plus a regression guard. Two blocks change (`poi-geocode`, `exports`); no contract changes.

- **`poi-geocode`**: `cleanPoiAddress` rewritten. The old version stripped a secondary-unit fragment (`Suite`/`Room`/`#`…) only when it was its **own comma-part**, so a unit packed **into the street segment with no comma** — `118 North Clark Street Room 230 Chicago`, `455 Golden Gate Ave. Suite 14800`, `88-11 Sutphin Blvd #106` — slipped through, the geocode returned nothing, and the office pin silently dropped. This is the shape every past incident took (and it recurred the moment a new roster source formatted addresses that way). The cleaner now removes the unit fragment **wherever it sits**, with two guards the old code lacked:
  - the fragment must be preceded by street content (a `(\S)` capture), so a **leading** primary number like `#10 Public Square` — a real county-clerk address — is never mistaken for a unit and stripped;
  - the token set drops `fl`/`no`, which collided with the **FL** state code (`Miami, FL 33131` → `Miami`) and ordinals; `floor` as a full word is retained and safe.
  This lives in the engine on purpose: fixing it here fixes every fork and every future data source, replacing the fragile per-builder "insert a comma" workaround (which is why the bug kept returning). Verified against all 101 real Illinois county-clerk addresses + the fleet's NYC/SF office addresses + the FL/leading-`#`/PO-box traps.
- **`exports`**: `cleanPoiAddress` is added to the debug namespace (`window.<Metro>Explorer`) so the smoke test can regression-guard the cleaner against a fixed case table **without any network call** — the durable protection against a future edit silently re-breaking it. Additive; no existing export changes.
- Consume-side scripts (`apply_engine.py`, `check_engine_parity.py`, `generate_metro_files.py`, `metro-worksheet.schema.json`) are unchanged.
- Fork adoption: nothing to do — both changes are inside the engine. Forks that carried the data-side comma-insertion workaround (SF's `build_ca_legislature_roster.py` `normalize_address`) can keep it as harmless defense-in-depth or drop it; the engine now handles the embedded-unit case regardless.
