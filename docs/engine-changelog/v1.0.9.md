Marker/region-highlight chrome fenced into the engine, plus a `districtPrefix` chamber-factory opt. Three new ENGINE blocks and one changed block; no contract breakage (both changes are additive/backward-compatible).

- **New blocks `styles-markers` / `map-chrome-classes` / `map-pan-filter`** (docs/ENGINE_SYNC.md item 10): the shared map-chrome classes were fork-prefixed (`chi-`/`nyc-`/`sf-region-highlight`, `-poi-pin`, `-panning`) with their CSS sitting unfenced beside fork content, so that region could never be byte-identical across forks. They are now neutralized to `region-highlight` / `poi-pin` / `map-panning` and fenced three ways: the CSS trio (`styles-markers`), the JS class constants — `HIGHLIGHT_CLASS` / `POI_PIN_CLASS` / `PANNING_CLASS`, promoted out of the worksheet + generator + schema into byte-identical engine constants (`map-chrome-classes`), and the `movestart`/`moveend` drop-shadow-pause toggle (`map-pan-filter`). Reconciling merged features rather than overwriting, so the forks that lacked the pan-pause optimization gained it.
- **Changed block `chamber-factory`**: new backward-compatible `districtPrefix` opt. It state-qualifies the district-number header (e.g. `IL-7`) for chambers whose districts read that way — used by the Chicago congress card once it moved onto the factory. Chambers that pass no prefix (the ILGA/other chambers) keep bare district numbers, so the change is inert for every existing registration.
- Consume-side scripts (`apply_engine.py`, `check_engine_parity.py`, `generate_metro_files.py`, `metro-worksheet.schema.json`) are unchanged.
- Fork adoption: nothing to do — all changes are inside the engine and arrive via the pinned bundle. Each fork's `engine.lock.json` bumps to `engine-v1.0.9` through the standard fan-out; the marker CSS-class rename and the `map-chrome-classes` constants replace whatever the fork's stylesheet/worksheet previously defined, so a fork's `smoke_test.mjs` must reference `.region-highlight` (all three siblings already do as of this release).
